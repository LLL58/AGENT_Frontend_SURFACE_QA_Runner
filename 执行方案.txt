
前端表层 Bug 自动巡检工具
Surface QA Agent Runner

它的核心目标是：

agent 运行一个命令
  ↓
工具自动打开页面、识别控件、执行低风险操作
  ↓
收集前端报错、接口错误、白屏、按钮异常、loading 卡死
  ↓
输出结构化 JSON 反馈
  ↓
agent 根据反馈修复代码
  ↓
agent 重新运行工具验证

Playwright 适合作为底层执行器，因为它支持浏览器自动化、页面事件监听、截图、trace、reporter、CLI 测试运行等能力；官方也支持同时使用多个 reporter，例如终端输出加 JSON 文件输出。 Playwright 的 Page 也可以监听页面事件，网络请求的 request / response / requestfailed 生命周期适合做前端异常采集；注意 HTTP 404/503 仍然是 HTTP 层面“完成的响应”，所以必须监听 response.status()，不能只监听 requestfailed。 Trace Viewer 可以查看执行 timeline、DOM 快照、网络请求，非常适合作为失败证据给人和 agent 分析。

1. 方案评估
1.1 能解决的问题

第一版可以稳定检查：

页面白屏
页面 JS runtime error
console.error
接口 4xx / 5xx
request failed
按钮无法点击
按钮点击后前端报错
按钮点击后接口报错
链接跳转 404
表单无法输入
loading 卡死
locator 失效
危险按钮误触发风险
1.2 不能解决的问题

它不能自动判断复杂业务正确性，例如：

订单金额是否算对
权限规则是否完全正确
复杂审批流是否符合预期
保存后的数据是否完全符合业务语义
UI 设计是否美观
文案是否产品满意

所以第一版不要追求“全业务自动测试”，而是做：

表层错误发现
结构化错误输出
agent 可复现修复
1.3 最终判断

这个方案可行，且值得做。
关键不是“自动点得多聪明”，而是错误反馈必须结构化、可复现、可被 agent 读取。

7. 执行定义
7.1 命令定义
{
  "scripts": {
    "surface:agent": "tsx tests/surface/run-agent-surface-check.ts",
    "surface:agent:ci": "tsx tests/surface/run-agent-surface-check.ts --ci",
    "surface:health": "tsx tests/surface/run-agent-surface-check.ts --health-only",
    "surface:route": "tsx tests/surface/run-agent-surface-check.ts --route",
    "surface:debug": "tsx tests/surface/run-agent-surface-check.ts --headed --debug"
  }
}

Playwright 官方 CLI 支持通过 npx playwright test 运行测试、指定文件、headed/debug/project 等参数；这里我们额外封装 tsx runner，是为了让 agent 拿到更稳定的自定义 JSON 输出。

7.2 CLI 参数
--ci
  CI 模式，有 error/critical 时返回非 0

--health-only
  只检查页面健康，不点击控件

--route=/profile
  只检查某个 route

--max-routes=10
  限制最多检查页面数

--max-controls=20
  限制每页最多执行控件数

--allow-warning
  允许执行 warning 动作

--headed
  有头浏览器模式

--debug
  调试模式

--output=.agent-feedback/run-001
  指定输出目录

7.4 执行流程
1. 读取 config
2. 创建 outputDir
3. 启动 browser/context/page
4. 开启 trace
5. attach error collector
6. 登录或加载 storageState
7. 遍历 routes
8. 打开页面
9. 执行 page health check
10. 扫描控件
11. 过滤 danger 控件
12. 逐个执行 safe 动作
13. 每个动作后收集错误 snapshot
14. 生成 issue
15. 保存 artifact
16. 写 stdout JSON
17. 停止 trace
18. 写 run-summary.json
19. 根据 severity 设置 exit code

9. 错误检查规则定义
9.1 页面级错误
检查项	判断	severity
页面白屏	body 文本极少且可见元素极少	critical
页面 404/500	主文档响应状态异常	critical
pageerror	未捕获 JS 异常	critical
console.error	控制台错误	error
requestfailed	网络请求失败	error
API 5xx	接口服务异常	critical
API 4xx	参数、权限、路由问题	warning/error
9.2 控件级错误
检查项	判断	severity
click timeout	按钮无法点击	error
locator error	定位器失效或多匹配异常	error
stuck loading	点击后长期 loading	error
unexpected navigation	跳到 logout、payment、delete 等危险路径	critical
no observable effect	点击后 URL、DOM、请求、弹窗都无变化	warning
action 后 pageerror	点击后页面崩溃	critical
action 后 network 5xx	点击后接口异常	critical
10. 退出码定义
0 = passed / passed_with_warnings
1 = 发现 error 或 critical
2 = 工具自身异常
3 = 测试环境不可用
4 = 配置错误

CI 模式建议：

if (summary.status === 'failed') {
  process.exitCode = 1;
} else if (summary.status === 'tool_error') {
  process.exitCode = 2;
} else if (summary.status === 'environment_error') {
  process.exitCode = 3;
} else {
  process.exitCode = 0;
}

agent 逻辑：

exit 0:
  不需要修复，或者只有 warning

exit 1:
  读取 issues.ndjson，开始修复

exit 2/3/4:
  优先检查工具、环境、配置，不要直接改业务代码
11. Agent 消费协议

agent 每次运行后只需要遵守这个协议：

1. 运行 npm run surface:agent
2. 读取 stdout 中 surface-qa.summary
3. 如果 status=passed，结束
4. 如果 status=failed：
   读取 .agent-feedback/latest/run-summary.json
   读取 .agent-feedback/latest/issues.ndjson
5. 按 severity 排序：
   critical > error > warning > info
6. 对每个 issue：
   查看 route
   查看 action
   查看 message
   查看 console/page/network errors
   查看 screenshot/html/trace
7. 定位代码修复
8. 重新运行 npm run surface:agent
9. 直到 summary.status=passed

13. 最终结论

你的最终工具应该是一个 Agent-first Frontend Surface QA Runner。

它不是普通测试报告工具，而是专门给 agent 设计的：

可执行
可复现
可结构化读取
可定位错误
可保存证据
可用退出码判断成功失败
可被 agent 反复运行验证

核心交付物就是这四个：

npm run surface:agent
.agent-feedback/latest/run-summary.json
.agent-feedback/latest/issues.ndjson
.agent-feedback/latest/artifacts/

只要这四个稳定，后面无论接 GitHub Issue、Jira、修复 agent、CI 还是自动 PR，都会非常顺
